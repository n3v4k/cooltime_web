# 쿨타임 (CoolTime) — PRD

## 1. 한 줄 요약
체감온도에 따른 폭염 작업중지·휴식 의무 기준을 실시간으로 확인하고, 작업자 개인은 물론 현장 관리자가 여러 작업자의 휴식 상태를 관리·기록할 수 있는 웹 서비스.

---

## 2. 문제 정의

> 2025년 7월 17일 산업안전보건기준에 관한 규칙 개정, 2026년 5월 대책 강화로 체감온도 31/33/35/38℃ 4단계 작업중지·휴식 기준이 시행 중이지만, 현장 근로자와 관리자는 "지금이 그 기준을 넘었는지, 얼마나 쉬어야 하는지"를 실시간으로 확인할 도구가 없다.

### 핵심 근거
- **법적 근거**: 산업안전보건기준에 관한 규칙 제558~562조 (2025.7.17 시행) — 31℃ 이상 2시간 연속작업 시 조치 의무, 33℃ 이상 2시간마다 20분 휴식 의무, 35℃ 이상 1시간마다 15분 휴식 + 오후 2~5시 옥외작업 중지 권고, 38℃ 이상 옥외작업 전면중지 권고
- **실제 처벌 사례**: 2025.6.13 대전지방법원 — 폭염 중 근로자 열사병 사망 사건에서 안전보건관리자는 산업안전보건법 위반, 원청 대표이사는 중대재해처벌법 위반으로 처벌. 사유는 휴식시간·그늘막 미제공, 물·소금 미비치
- **통계 근거**: 온열질환 산재 승인 건수 2020년 13건 → 2025년 77건 (5.9배 증가), 2025년 5월까지 사망 승인만 4명, 최근 5년 산재 228명 중 건설업이 46.5%
- **구조적 한계**: 사업주는 폭염작업 시 체감온도·조치사항을 기록·보관할 법적 의무가 있으나, 표준화된 기록 도구가 없어 수기 관리에 의존

## 2-1. 왜 이 해결책이 통하는가 (Solution-Problem Fit)

앞의 문제 정의가 "왜 심각한가"였다면, 이 섹션은 "왜 하필 실시간 확인 도구가 답인가"에 대한 근거다.

**① 병목은 태만이 아니라 정보 부족**
윌로그의 사업장 대상 설문조사에서 응답자의 54%가 체감온도 기준 예방조치·기록보관 의무 신설 사실을 인지하지 못했고, 20%만이 폭염 판단 기준이 체감온도임을 알고 있었다. 즉 문제의 핵심은 "몰라서 못 지킨다"는 인지 격차이며, 이는 실시간 정보 제공 도구가 정확히 해결할 수 있는 유형의 문제다.

**② 유사 개입의 실증 사례**
기상청의 '폭염 영향예보 직접전달 서비스'(농촌 취약계층 대상, 2022~)는 2024년 여름 서비스 대상자의 온열질환 피해 0건, 만족도 95~98%를 기록했다. 복잡한 기상 기준을 개인이 이해하기 쉬운 형태로 실시간 전달하는 것만으로 실제 피해가 줄어든 선례다. (단, 단일 지역·소규모 시범사업 결과이며 무작위 대조 실험은 아님을 유의)

**③ 업계도 같은 방향으로 투자 중 (시장 검증)**
롯데건설은 전국 80개 현장에 IoT 기반 체감온도 실시간 모니터링 플랫폼을 가동 중이며, 삼성전자는 갤럭시 워치로 체온·환경을 실시간 분석해 휴식 권고 알림을 보내는 산업안전 솔루션을 출시했다. "실시간 체감온도 확인 + 알림"이라는 접근 자체는 이미 검증된 방향이다.

**④ 국제 학계도 같은 갭을 지적**
Edirisinghe & Gunathilake(2025, *International Journal of Construction Management*)는 건설업 열 스트레스 관리의 기존 시스템들이 실시간 모니터링·개인화·전력 효율성이 부족해 효과가 제한적이라고 지적하며 IoT 스마트 조끼를 대안으로 제시했다. 즉 "실시간성 + 개인화 부족"은 학계에서도 공인된 갭이며, 쿨타임은 고가 IoT 장비 없이 이 갭을 스마트폰 웹으로 메운다.

**쿨타임의 포지셔닝**: 위 사례들은 IoT 센서나 스마트워치 등 인프라 투자가 필요한 대기업 전용 솔루션인 반면, 쿨타임은 스마트폰 웹 하나로 동일한 핵심 가치(실시간 기준 확인 + 휴식 알림)를 제공해 인프라 투자가 어려운 중소 현장과 개인 작업자까지 커버한다.

## 4. 핵심 기능 (MVP 범위)

### 4.1 개인 모드
- 위치 기반 현재 체감온도 표시
- 4단계(정상/주의/경고/위험/매우위험) 신호등 배지
- 해당 단계별 휴식 규칙 + 카운트다운 타이머
- 옥외/실내 작업 토글 (오후 2~5시 특례 반영)
- 온열질환 종류별 증상·대처법 (섹션 8 참고)
- **온열질환 자가진단 (3문항)**: 이상 증상 발생 시 예/아니오 3문항으로 유형 추정 → 대처법 표시, 열사병 의심 시 즉시 119 안내
- **현장 참여**: 4자리 현장 코드 입력으로 관리자와 연결
- 법적 근거 요약 (접기/펼치기)

### 4.2 관리자 모드
- 현장 공통 체감온도 + 등급 표시
- 전체 작업자 요약 카드 (작업 중 / 휴식 중 / 휴식 초과 인원수)
- 휴식 초과자 경고 배너
- **온열질환 의심 알림 배너**: 작업자가 자가진단에서 의심 소견을 남기면 실시간으로 최상단에 긴급 표시
- 작업자별 리스트: 이름, 개인 상태, 남은 시간, 휴식 시작 버튼 (다른 기기의 작업자가 직접 누른 경우도 실시간 반영)
- 작업자 추가 (이름 입력) + 현장 코드 발급
- **오늘 기록 CSV 내보내기** (일자·시각·작업자·체감온도·등급·이벤트·의심 온열질환 유형)

## 5. 판정 로직

| 단계 | 체감온도 | 조치 | 성격 |
|---|---|---|---|
| 정상 | 31℃ 미만 | 없음 | - |
| 주의 | 31~32.9℃ (2시간+) | 냉방/시간조정/휴식 중 1개 이상 | 법적 의무 |
| 경고 | 33~34.9℃ | 2시간마다 20분 휴식 | 법적 의무 |
| 위험 | 35~37.9℃ | 1시간마다 15분 휴식 + 옥외는 오후 2~5시 중지 권고 | 권고 |
| 매우위험 | 38℃ 이상 | 옥외작업 전면중지 (긴급조치 제외) | 강력 권고 |

**상태 전이 규칙**: 상위 단계로 오를 때만 휴식 타이머 리셋(보수적 해석). 하위 단계로 내려가도 타이머는 유지하고 규칙만 전환.

### 5-1. 온열질환 자가진단 로직

```
Q1. 의식이 흐릿하거나 없나요?
 └ 예 → 열사병 의심 (최고 위험) → 즉시 119 신고 화면 + 관리자 긴급 알림
 └ 아니오 → Q2

Q2. 피부가 차갑고 축축한가요, 아니면 근육 경련이 있나요?
 └ 차갑고 축축함 → 열탈진 의심
 └ 근육 경련 → 열경련 의심
 └ 둘 다 아니면 → Q3

Q3. 서있다가 갑자기 어지러워 쓰러졌나요, 아니면 손발이 부었나요?
 └ 쓰러짐 → 열실신 의심
 └ 부종 → 열부종 의심
 └ 해당 없음 → 유형 특정 어려움 → 증상표 참고 안내 + 경과 관찰 권고
```

질문은 3개로 제한해 응답 부담을 최소화하고, 열사병 의심 시에는 나머지 질문 없이 즉시 119 화면으로 전환한다. 결과는 대처법 표시와 함께 로그에 자동 기록되고, 관리자 화면에도 실시간 반영된다.

## 6. 데이터 모델 (Supabase)

작업자와 관리자가 서로 다른 기기를 쓴다는 전제로, 상태를 기기가 아닌 서버(Supabase)에 둔다. 관리자 한 명이 여러 작업자를 실시간으로 봐야 하므로, "같은 현장에 속해 있다"는 것을 식별할 키가 필요하다 → **4자리 현장 코드**로 해결한다 (로그인/회원가입 없이 코드만 입력하면 같은 현장 데이터에 접근).

```sql
-- 현장 (관리자가 생성, 4자리 코드로 작업자 참여)
sites (
  id uuid primary key default gen_random_uuid(),
  join_code text unique,        -- 예: "3821"
  name text,
  feels_like_temp numeric,      -- 현장 공통 체감온도 (관리자 기기가 주기적으로 갱신)
  tier text,                    -- 정상/주의/경고/위험/매우위험
  created_at timestamptz default now()
)

-- 작업자 (개인 모드에서 코드 입력 시 자동 생성/연결)
workers (
  id uuid primary key default gen_random_uuid(),
  site_id uuid references sites(id),
  name text,
  is_outdoor boolean,
  tier_entered_at timestamptz,
  rest_status text,             -- 'working' | 'resting' | 'overdue'
  rest_started_at timestamptz,
  updated_at timestamptz default now()
)

-- 이벤트 로그 (기록·CSV 추출용)
logs (
  id uuid primary key default gen_random_uuid(),
  site_id uuid references sites(id),
  worker_name text,
  timestamp timestamptz default now(),
  site_temp numeric,
  tier text,
  event_type text,   -- 'REST_START'|'REST_END'|'OVERDUE'|'TIER_UP'|'ILLNESS_REPORTED'
  note text          -- 온열질환 의심 유형 등 ('열사병'|'열탈진'|'열경련'|'열실신'|'열부종'|null)
)
```

- 온도(tier)는 `sites` 테이블에 1행만 존재 (현장 공통), 작업자별 타이머 상태는 `workers`에 개별 관리
- 온열질환 자가진단 결과는 `logs`에 `ILLNESS_REPORTED` 이벤트로 적재되고 `note`에 의심 유형 저장
- **동기화 방식**: Supabase Realtime(웹소켓) 대신, 데모 안정성을 위해 5~10초 간격 폴링(`setInterval`로 재조회)으로 MVP를 시작. 이후 필요 시 Realtime 구독으로 전환
- **보안**: RLS(Row Level Security)로 같은 `site_id`를 가진 행만 조회/수정 가능하도록 최소한의 정책만 설정 (계정 시스템 없이 코드 기반 접근 제어)

## 7. 기술 방향
- **날씨 데이터**: OpenWeatherMap API (기온+습도 → 체감온도 계산, 관리자 기기가 주기적으로 조회해 `sites` 테이블 갱신)
- **상태 저장/동기화**: **Supabase** (Postgres + 간단한 RLS) — 작업자·관리자가 서로 다른 기기를 쓰는 구조를 지원하기 위해 `localStorage` 대신 채택
- **인증**: 계정 시스템 없이 4자리 현장 코드로 접근 (MVP 범위에서 회원가입/로그인 생략)
- **기록 내보내기**: `logs` 테이블을 CSV로 export (Blob + 다운로드 링크), 확장 시 Excel/PDF 검토

## 8. 온열질환 종류 및 대처법 (교육 콘텐츠 섹션)

법적 기준 확인 기능 외에, 실제 증상이 나타났을 때 참고할 수 있도록 질병관리청·안전보건공단 자료 기반의 대응 정보를 앱 내에 함께 제공한다. (개인 모드 하단 또는 별도 탭으로 배치)

| 종류 | 주요 증상 | 대처법 |
|---|---|---|
| **열사병** (가장 위험) | 체온조절 중추 기능 상실로 체온이 40℃ 이상 치솟음, 피부가 건조하고 뜨거움, 의식장애·혼수 동반 가능 | 즉시 119 신고 + 시원한 장소로 이동, 옷을 헐렁하게 하고 몸을 물수건 등으로 식히며 병원 이송이 최우선 |
| **열탈진** (일사병) | 체온 37~40℃, 땀을 많이 흘리고 피부는 차갑고 축축하며 창백함, 두통·어지럼증·메스꺼움·무기력감 | 그늘/에어컨이 있는 곳으로 옮겨 눕히고 다리를 높게, 의식이 있고 구토가 없으면 물·이온음료 섭취, 1시간 내 회복 안 되면 병원 이송 |
| **열경련** | 땀으로 염분·칼륨·마그네슘이 손실되어 종아리·허벅지·어깨 등 근육경련 발생 | 시원한 곳에서 휴식, 수분·이온음료 보충, 경련 부위 마사지 (경련이 멈춰도 바로 작업 복귀 금지, 충분히 휴식) |
| **열실신** | 체표면 혈액이 늘고 심부 혈액량이 줄어 뇌 혈류 부족으로 일시적 의식소실, 오래 서 있다 갑자기 일어날 때 주로 발생 | 그늘에 눕히고 다리를 심장보다 높게, 의식 회복 확인 후 충분한 휴식과 수분 보충 |
| **열부종** | 더운 환경에서 손·발 등이 붓는 비교적 경미한 증상 | 부은 부위를 심장보다 높게 올리고 휴식, 증상이 지속되면 병원 진료 |

**공통 판단 기준**: 열탈진·열경련 증상이 1시간 이상 지속되거나 회복되지 않으면 반드시 병원 진료를 받는다. 열사병 의심 시(의식저하, 고체온, 건조한 피부)에는 지체 없이 119에 신고한다.

> 본 섹션은 참고용 안전 정보이며, 실제 응급 상황에서는 전문 의료기관의 진단과 처치를 최우선으로 한다.

## 9. 화면 설계 및 플로우 (다이어그램 설명)

### 9-1. 데이터 흐름도 (Supabase 연결 구조)
작업자 기기(기기 A)와 관리자 기기(기기 B)가 서로 다른 물리적 기기임을 전제로 한 흐름이다.
```
[작업자 기기] --동기화--> [Supabase (sites·workers·logs)] --폴링 조회(5~10초)--> [관리자 기기]
```
- 작업자 기기: 현장 코드로 참여, 본인 상태(휴식 시작/종료, 자가진단 결과)를 Supabase에 기록
- Supabase: 3개 테이블(`sites`, `workers`, `logs`)이 현장 단위로 데이터를 묶는 허브 역할
- 관리자 기기: 여러 작업자의 상태를 5~10초 간격 폴링으로 조회해 대시보드에 반영 (Realtime 구독은 향후 확장)

### 9-2. 온열질환 자가진단 플로우차트
3개의 예/아니오 질문으로 5가지 유형(열사병/열탈진/열경련/열실신/열부종) 중 의심 소견을 추정한다.
```
이상 증상 발생
  └ Q1. 의식이 흐릿하거나 없나요?
      ├ 예 → 열사병 의심 (최고 위험) → 즉시 119 신고
      └ 아니오 → Q2. 주요 증상이 있나요?
          ├ 차갑고 축축함 → 열탈진 의심
          ├ 근육 경련 → 열경련 의심
          └ 둘 다 아니면 → Q3. 쓰러지거나 부었나요?
              ├ 쓰러짐 → 열실신 의심
              ├ 부종 → 열부종 의심
              └ 해당 없음 → 유형 미확정, 경과 관찰 권고
```
색상 의미: 빨강(열사병)=응급 대응 필요, 주황(나머지 4개 유형)=주의 및 대처법 안내, 회색=진행 중인 질문 단계. 열사병 의심 시에는 나머지 질문 없이 즉시 119 화면으로 전환된다.

### 9-3. 작업자 화면 와이어프레임
- 상단: 위치 + 현장 코드(예: 3821) + 동기화 상태
- 중앙: 체감온도 큰 숫자 + 단계 배지(색상)
- 행동 카드: 휴식 규칙 문구 + 카운트다운 타이머 + "휴식 타이머 시작" 버튼
- **"몸이 안 좋아요 · 자가진단" 버튼**: 빨간 톤으로 강조, 탭 시 9-2 플로우 진입
- 하단: 옥외/실내 작업 토글

### 9-4. 관리자 대시보드 와이어프레임
- 상단: 현장 코드 + 공통 체감온도 + 단계 배지
- 요약 카드 3개: 작업 중 / 휴식 중 / 휴식 초과 인원수
- **온열질환 의심 긴급 배너** (최상단, 휴식 초과 배너보다 우선순위 높게 배치): "OOO 님 열사병 의심 · 즉시 확인하세요" + "119 안내" 버튼
- 휴식 초과 경고 배너: 초과자 이름 나열
- 작업자 리스트: 이름, 상태, 남은 시간, 상태별 액션 버튼 (초과자는 빨간 테두리로 구분)
- 하단: "작업자 추가" + "오늘 기록 내보내기(CSV)" 버튼

## 10. 브랜드 아이덴티티
- **이름**: 쿨타임 (게임의 '쿨다운' + '더위를 식히는 시간'의 중의적 표현)
- **컨셉**: 밝은 하늘색 배경의 공익 캠페인 톤, 시계 다이얼에 온도계 모양의 시침·분침(코랄=시침, 파랑=분침)을 결합한 로고
- **산출물**: 커버 이미지(380×270), 로고 이미지(150×100)

## 11. 성공 지표 (제안)
- 체감온도 확인~휴식 타이머 시작까지 소요 시간 3초 이내
- 관리자가 초과자를 인지하는 데 걸리는 시간 (경고 배너 노출 여부로 측정)
- 정성 지표: 사전 설문으로 "법 개정 인지도가 낮다"는 문제 가설 검증

## 12. 향후 확장 (Out of Scope)
- Supabase Realtime(웹소켓) 기반 실시간 구독으로 전환 (MVP는 폴링 방식)
- 계정/로그인 시스템 도입 (MVP는 현장 코드 기반 무인증 접근)
- 지역별 온열질환 통계 지도 시각화
- 신규 배치자 대상 열순응 안내 (산재 사망자 80.6%가 투입 7일 이내 발생)
- Excel/PDF 형태의 정식 기록 리포트
